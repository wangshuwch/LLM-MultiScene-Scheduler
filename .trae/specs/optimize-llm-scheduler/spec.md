# LLM 多场景调度器优化 Spec

## Why

当前实现已初步搭建了 LLM 多场景调度系统，但需要进一步优化以确保：
1. 资源足够时，场景能充分利用资源且不受场景级上限限制
2. 资源不足时，严格按场景优先级分配资源
3. 队列调度算法明确且可验证
4. 调度流程更高效（先检查资源可用性再从队列中取出请求）

## What Changes

### 核心优化点

1. **资源管理器优化**
   - 修复 `_calculate_total_demand` 逻辑（当前仅返回已用 token，未包含队列中等待的 token）
   - 明确"资源充裕"和"资源紧张"的判断标准
   - 优化超卖逻辑

2. **队列调度优化**
   - 实现更明确的队列选择算法（优先级 + 入队时间）
   - 添加资源预检查机制：先判断是否有可用资源，再从队列中选择合适的请求
   - 确保同优先级场景按 FIFO 顺序调度

3. **调度器流程优化**
   - 优化 `_try_dispatch` 逻辑，避免资源获取失败后重新入队
   - 提高调度效率

### 队列设计与消费规则详解

#### 队列数据结构
- **每个场景独立队列**：`scene_id -> deque(LLMRequest)`
- **队列属性**：
  - `max_size`：队列最大长度
  - `waiting_tokens`：队列中所有请求的预估 token 总和

#### 消费规则（调度算法）
```
1. 收集所有非空队列的队首请求
2. 按以下优先级排序：
   a. 一级：场景优先级（数字越小优先级越高）
   b. 二级：请求入队时间（越早越优先）
3. 按排序后的顺序，依次尝试为请求分配资源：
   a. 检查全局可用 token 是否足够
   b. 检查场景当前使用量 + 请求 token 是否超过场景级上限（仅当资源紧张时）
   c. 第一个满足条件的请求被选中执行
4. 如果没有请求能获取到资源，等待下一轮调度
```

#### 资源紧张判断标准
- **资源充裕**：`已用并发 token + 队列中等待的 token 总和 ≤ 总容量`
  - 场景可突破自身 `max_concurrent_tokens` 限制
- **资源紧张**：`已用并发 token + 队列中等待的 token 总和 > 总容量`
  - 场景受自身 `max_concurrent_tokens` 限制

## Impact

- 受影响的模块：`resource_manager.py`, `queue_manager.py`, `scheduler.py`
- 受影响的测试：`test_scheduler.py`

## ADDED Requirements

### Requirement: 优化的资源分配策略
系统 SHALL 根据资源充裕/紧张状态采用不同的资源分配策略。

#### Scenario: 资源充裕时
- **WHEN** 所有场景当前请求的总资源需求 ≤ LLM 总能力
- **THEN** 每个场景都能尽可能拿到自己所需资源，不受场景级 `max_concurrent_tokens` 限制

#### Scenario: 资源紧张时
- **WHEN** 所有场景当前请求的总资源需求 > LLM 总能力
- **THEN** 按照预设的场景优先级分配资源，高优先级场景优先获得资源，低优先级场景排队等待

### Requirement: 明确的队列调度规则
系统 SHALL 按照明确的规则从队列中选择下一个请求执行。

#### Scenario: 队列调度
- **WHEN** 有请求在队列中等待
- **THEN** 系统按以下规则选择请求：
  1. 高优先级场景的请求优先
  2. 同优先级场景中，入队时间早的请求优先

## MODIFIED Requirements

### Requirement: 调度流程
原调度流程先从队列取出请求，再尝试获取资源，失败后重新入队。修改为：先检查资源可用性，再选择合适的请求取出并执行。
